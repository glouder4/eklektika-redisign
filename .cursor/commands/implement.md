# implement

Запусти процесс разработки строго по `@.cursor/skills/composite-subagent-workflow/SKILL.md`.

Вход: задача пользователя из текущего сообщения.

## Обязательный способ исполнения (не подменять)

Пока пользователь вызвал **эту** команду (`/implement`), основной агент **обязан** реально вызывать субагентов через инструмент **`Subagent`** с полем `subagent_type`, а **не** воспроизводить роли тимлида/разработчика одним своим ответом и не ограничиваться чтением `@.cursor/agents/*.md` без вызова `Subagent`.

| Шаг | Как вызывать |
|-----|----------------|
| Постановка и риски | `Subagent` с `subagent_type`: **`composite-teamlead`**, в `prompt` — задача, ограничения, ожидаемый результат; требуй структуру из SKILL (риски, план, DoD, вопросы). |
| Реализация | `Subagent` с `subagent_type`: **`composite-developer`**, в `prompt` — полный вывод тимлида как обязательная рамка. |
| Ревью | снова `Subagent` с **`composite-teamlead`**: исходная задача, план/DoD, отчёт и изменения разработчика. |
| Цикл | при замечаниях — снова `composite-developer`, затем ревью тимлидом, пока не выполнен критерий остановки из SKILL. |

**Запрещено:** считать шаг выполненным, если субагент не был вызван через `Subagent` (в т.ч. для «только аудита», «только вопроса» или мета-задачи — тогда тимлид всё равно через `Subagent`, разработчик — если по смыслу нужен разбор кода/диффа).

**Допустимо без второго субагента:** только если тимлид явно зафиксировал, что **кода менять не нужно** и достаточно его постановки — тогда второй шаг: краткий вызов `Subagent` к `composite-developer` с запросом «только подтвердить отсутствие правок / перечислить затронутые файлы» или один проход тимлида с явным DoD «без PR»; не пропускай `Subagent` полностью.

Обязательный порядок шагов (содержательно тот же, что в SKILL):

1. Сначала **`Subagent` → `composite-teamlead`** (контекст: `@.cursor/agents/composite-teamlead.md`) — задача, риски, план, Definition of Done.
2. Затем **`Subagent` → `composite-developer`** (контекст: `@.cursor/agents/composite-developer.md`) — реализация по рамке.
3. После реализации снова **`Subagent` → `composite-teamlead`** — ревью качества.
4. Если есть замечания — снова **`composite-developer`**, затем ревью; повторять до критерия остановки из SKILL.

Важно:

- Перед постановкой и реализацией обязательно учитывай `@.cursor/rules/arch-rules.mdc`.
- В каждом цикле требуй явный блок по документации:
  - какие документы в `docs/features/` изучены;
  - какие документы обновлены;
  - что ещё нужно документировать.
- Запуск тестов выполняет пользователь вручную.
- Агент не запускает тесты автоматически и в конце каждой итерации обязан явно дать пользователю список проверок для ручного запуска.

Финальный ответ пользователю (после всех вызовов `Subagent`):

- что реализовано;
- какие замечания тимлида были исправлены;
- какие изменения внесены в документацию (`docs/features/`);
- какие ручные тесты должен запустить пользователь;
- кратко: какие вызовы `Subagent` (роли субагентов) были выполнены по шагам.
