# orchestrate

Запусти процесс разработки комплексной задачи по `@.cursor/skills/implementation-orchestration/SKILL.md`.

Вход: задача пользователя из текущего сообщения.

## Обязательный способ исполнения (не подменять)

Пока пользователь вызвал **эту** команду (`/orchestrate`), основной агент **обязан** вызывать субагентов через инструмент **`Subagent`** с корректным `subagent_type`. **Запрещено** выполнять роль планировщика, тимлида или разработчика одним ответом без соответствующих вызовов `Subagent`.

| Этап | Как вызывать |
|------|----------------|
| Планирование (комплексная задача или список задач) | `Subagent` с `subagent_type`: **`implementation-planner`** — передавай формулировку пользователя, контекст, критерии готовности; планировщик создаёт артефакты по SKILL в `docs/tasks/…`. |
| Реализация | после планировщика **обязательно** выполни `@.cursor/commands/implement.md`, то есть цепочку **`Subagent` → `composite-teamlead` → `composite-developer` → `composite-teamlead`** (и итерации) как в команде implement. |
| Маленькая задача | если декомпозиция не нужна, **не** вызывай `implementation-planner`, но **всё равно** запускай **`@.cursor/commands/implement.md`** — то есть полный цикл `Subagent` (тимлид → разработчик → тимлид …). |

**Запрещено:** создавать папки `docs/tasks/…` и файлы задачи самостоятельно вместо прогона **`Subagent` → `implementation-planner`**, если по смыслу сработала комплексная ветка (список задач, крупная декомпозиция).

При необходимости уточнений от планировщика сначала задай вопросы пользователю, затем снова `Subagent` → **`implementation-planner`** (или продолжай с уточнённым контекстом), затем implement.

Обязательная логика (содержательно):

1. Определи тип задачи:
   - если комплексная или список задач — сначала **`Subagent` → `implementation-planner`** (`@.cursor/agents/implementation-planner.md`);
   - если маленькая — сразу **`@.cursor/commands/implement.md`** (только через `Subagent`, см. выше).
2. Для комплексной задачи после завершения планировщика передай результат в **`@.cursor/commands/implement.md`** и выполни цепочку `Subagent` до критерия из SKILL implement.
3. При блокирующих вопросах к пользователю — пауза, ответ пользователя, затем продолжение с `Subagent`.
4. На этапе планирования и реализации обязательно контролируй документацию:
   - опора на `docs/features/README.md` и профильные `docs/features/*.md`;
   - фиксация плановых и фактических обновлений docs в каждом этапе.

Финальный ответ пользователю:

- что было запланировано (и что создал планировщик в `docs/tasks/…`, если применимо);
- что передано в реализацию;
- какие изменения внесены в документацию (`docs/features/`);
- какие ручные проверки нужно выполнить пользователю;
- кратко: перечисли выполненные вызовы `Subagent` по ролям (`implementation-planner` / `composite-teamlead` / `composite-developer`) по порядку.
