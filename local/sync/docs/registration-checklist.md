# Чеклист проверки регистрации (сайт + B24)

Документ для ручной/авто-проверки процесса регистрации пользователя и синхронизации с CRM.

## 1) Валидация формы (UI + backend)

- [ ] Обязательные поля не пустые: `name`, `lastname`, `main-phone`, `inn`, `name_company`, `password`.
- [ ] `inn` принимает только 10 или 12 цифр.
- [ ] `password` и `password_confirm` совпадают, длина пароля не менее 6.
- [ ] `email` (если заполнен) проходит формат.
- [ ] При ошибке пользователь получает понятный текст, без 500/HTML в JSON.

Ожидаемый результат:
- фронт блокирует некорректные значения;
- backend повторно валидирует и возвращает `success=false` + `error`.

## 2) Уникальность на сайте (пользователи + компания в ИБ)

### Пользователь (`b_user`)

- [ ] При существующем email регистрация не создает нового пользователя.
- [ ] При существующем телефоне регистрация не создает нового пользователя.
- [ ] В ответе: явная ошибка о дубликате (без «тихого» `CUser::Add`).

### Компания (`IBLOCK_ID=23`)

- [ ] По `INN` ищется существующий элемент компании.
- [ ] Если компания найдена, новый элемент ИБ **не создается**.
- [ ] Новый пользователь добавляется в `LEGAN_ENTITY_USERS` существующей компании.
- [ ] Если компания не найдена, создается новый элемент ИБ 23 с корректными свойствами.

Ожидаемый результат:
- нет дублей пользователей;
- нет дублей компаний по ИНН;
- при совпадении компании только допривязка пользователя в `LEGAN_ENTITY_USERS`.

## 3) Сайт -> B24: поиск компании по ИНН

- [ ] Сайт отправляет запрос в B24 через входящий endpoint портала (без login HTML в ответе).
- [ ] Если у локальной компании заполнен `OS_COMPANY_B24_ID`, все вызовы выполняются только через этот ID (`crm.company.get`), без поиска по ИНН.
- [ ] Поиск компании в CRM по ИНН (`crm.requisite.list`) используется только когда локальная связка `OS_COMPANY_B24_ID` отсутствует.
- [ ] Если компания найдена:
  - [ ] проверяется наличие контакта;
  - [ ] при отсутствии создается новый контакт и привязывается к найденной компании.
- [ ] При невалидном/неразрешимом `OS_COMPANY_B24_ID` регистрация завершается контролируемой sync-ошибкой (без fallback на ИНН).

Ожидаемый результат:
- если CRM-компания уже есть, дубли компании не появляются;
- контакт создается/привязывается корректно.

## 4) Проверка контакта в CRM

- [ ] Контакт ищется по email/телефону.
- [ ] Если контакт уже существует, новый не создается.
- [ ] Если контакта нет, создается новый.
- [ ] Для созданного/найденного контакта корректно выставляется связь с компанией.

Ожидаемый результат:
- в CRM нет дублей контактов;
- контакт связан с нужной компанией.

## 5) Минимальный smoke-набор сценариев

- [ ] Новый пользователь + новая компания (чистый кейс).
- [ ] Новый пользователь + существующая компания по ИНН.
- [ ] Дубликат email.
- [ ] Дубликат телефона.
- [ ] Повтор регистрации с тем же ИНН и другим пользователем (проверка `LEGAN_ENTITY_USERS`).

## 7) Критерии готовности

- [ ] Все сценарии smoke-набора проходят.
- [ ] Дубликаты пользователей/компаний/контактов не создаются.
- [ ] На шаге сайт -> B24 нет HTML login вместо JSON.
- [ ] Ошибки возвращаются в контролируемом JSON-формате.
- [ ] Контракт INN-first + B24-by-ID соблюден во всех ветках (успех/ошибка).
