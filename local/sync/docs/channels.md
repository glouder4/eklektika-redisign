# Каналы и транспорт

Предметная модель данных, ключи сущностей, удаления, полнота канала **сайт → CRM**, фасад `ajax.php`, поведение при сбоях и приём на портале — в **[functional-contract.md](functional-contract.md)**. Короткая шпаргалка на аварию (опционально) — **[runbook.md](runbook.md)**.

## Направления

1. **Сайт → CRM** (`to-crm` / исходящий)  
   Реакции на события сайта, отправка данных на портал Bitrix24 (REST и сценарии в коде).

2. **CRM → Сайт** (`from-crm` / входящий)  
   События на портале → HTTP на сайт → применение к сущностям сайта.

«Где сработало событие» и «куда текут данные» — разные вопросы; канал задаётся **направлением потока данных**.

## Транспорт CRM → сайт

| Элемент | Описание |
|---------|----------|
| Отправитель | Bitrix24, классы вроде `Updater::sendRequest` |
| Получатель | Сайт, `POST` на `/local/classes/ajax.php` |
| Диспетчеризация | Параметр `ACTION` (например `UPDATE_MANAGER`, `UPDATE_COMPANY`) |
| Базовый URL сайта для CRM | Константа вроде `EKLEKTIKA_SITE_URL` (окружения — из настроек, не хардкод прод без необходимости) |

Статус на 2026-04-24:
- `ajax.php` работает как тонкий endpoint и делегирует маршрутизацию в `local/sync/from-crm/InboundGateway.php`.
- Проверка входа централизована в `InboundSecurity` (токен через `X-Sync-Token`/`sync_token`, если задан секрет).
- Для удаления компании поток фиксирован: `OnBeforeCrmCompanyDelete` (B24) -> `ACTION=DELETE_COMPANY` -> удаление элемента в ИБ `23` на сайте.

## Транспорт сайт → CRM

| Элемент | Описание |
|---------|----------|
| С сайта | Запросы к API портала, на сайте ориентир `URL_B24` и обёртки в `local/classes/b24/` (до переноса в `sync/`) |

Статус на 2026-04-24:
- На стороне сайта B24-клиент и usersync-логика выносятся в модули `local/modules/eklektika.b24.rest` и `local/modules/eklektika.b24.usersync`.
- Временная совместимость обеспечивается через `local/classes/requires.php`, который подключает `include.php` модулей в фиксированном порядке.

## Что планируется при рефакторинге

- Единый каталог **`local/sync/`** для новой и переносимой логики; **не** добавлять новые разрозненные папки «под CRM» и «под сайт» на верхнем уровне `local/`.
- Подпись/токен для `ajax.php`, идемпотентность, политика конфликтов — по плану рефакторинга в Cursor Plans (файл вида `рефакторинг_синхронизации_b24_*.plan.md`).
- Документация обновляется здесь и в [eklektika-rules.mdc](../../../../../.cursor/rules/eklektika-rules.mdc).

## Сторона Bitrix24

В репозитории `eklektika-ru-b24` — зеркальная идея: **одна выделенная папка** под интеграцию с сайтом и **документация внутри неё** (не размазывать по `local/` без необходимости). Имя папки согласовать с командой (`sync/`, `integration/` и т.д.).
