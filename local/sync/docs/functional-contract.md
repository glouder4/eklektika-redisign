# Предметный контракт обмена (данные и поведение)

Дополняет [`channels.md`](channels.md) (каналы и транспорт). Здесь — **что** синхронизируем и **как** вести себя отправителям/приёмникам; без этого интеграция остаётся «папками и ACTION без смысла».

Заполнять по мере внедрения; пустые разделы в **§1–6** — явный долг для конвейера. §7 — по необходимости.

## 1. Идентичности и связи сущностей

Одного `ACTION` недостаточно: для каждой синхронизируемой сущности зафиксировать:

| Поле | Описание |
| ---- | -------- |
| Сущность | Имя в предметной области (пользователь, компания, …) |
| Ключ на стороне B24 | ID сущности в CRM, либо другой стабильный идентификатор из API |
| Ключ на стороне сайта | ID в БД сайта, логин, внешний код, составной ключ — что используется при **upsert** |
| Правило сопоставления | Как находить пару при первом обмене и при обновлениях (по email, по внешнему полю, только по переданному B24_ID, …) |

Черновик по коду (уточнять при изменении полей):

| Сущность | Ключ B24 | Ключ на сайте | Сопоставление |
| -------- | -------- | -------------- | ------------- |
| Контакт (CRM) / пользователь | ID контакта в B24 | `b_user.UF_BITRIX24_ID` (константа `UserSyncConfig::USER_UF_CONTACT_B24_ID`), запасно `UF_B24_USER_ID` | Поиск по `B24_ID` ↔ `User::getUserIDByB24ID` |
| Поля контакта CRM во входящем `UPDATE_CONTACT` | Ключи вида `UF_CRM_*` в POST | Поля `b_user` / побочная логика | Нормализация и коды полей — **`OnlineService\Sync\FromCrm\CrmInboundUfMap`** (`local/sync/from-crm/CrmInboundUfMap.php`); неизвестные `UF_CRM_*` перед `CUser::Update` снимаются, чтобы не передавать несуществующие ключи пользователю сайта. |
| Компания | `OS_COMPANY_B24_ID` / ID компании в B24 | элемент ИБ, `CODE` / свойства компании | `Company::getCompanyByB24ID` и обновление по `OS_COMPANY_B24_ID` |
| Менеджер (витрина) | `BITRIX24_ID` (user id в B24) | элемент ИБ менеджеров, `XML_ID` | `Manager::update` по `BITRIX24_ID` |
| Группы пользователей | данные в `UPDATE_GROUP` | `UserGroups` | см. `UserGroups` и параметры запроса |

**Конкуренция:** при одновременном изменении одной и той же сущности с сайта и из CRM действует **последняя успешно применённая запись** (явных версий в обмене сейчас нет — при необходимости добавить поле версии / timestamp в payload).

### Контракт регистрации (INN-first + локальная связка B24_ID)

- Сначала выполняется локальный lookup компании по ИНН в ИБ компании.
- Если локальная компания найдена и `OS_COMPANY_B24_ID` заполнен:
  - все операции в B24 выполняются только по этому ID (`crm.company.get`);
  - поиск компании в B24 по ИНН запрещён;
  - при невалидном/неразрешимом B24 ID регистрация завершается контролируемой ошибкой sync (без fallback на INN).
- Поиск B24 по ИНН (`crm.requisite.list` с `RQ_INN`) разрешён только при отсутствии локальной связи (`OS_COMPANY_B24_ID` не задан).
- Single-writer для локальной связки: в потоке регистрации запись/обновление `OS_COMPANY_B24_ID` выполняется только через `RegisterUserCompany::upsertSiteCompanyLinkByB24Id`.

## 2. Жизненный цикл: удаление / деактивация

| Тип | Удаление / отвязка в обмене | Идемпотентность |
| --- | --------------------------- | ---------------- |
| Контакт | `DELETE_CONTACT` → сайт; удаление пользователя на сайте тянет `OnBeforeUserDelete` → удаление контакта в B24 через `User` | Повтор `DELETE` желательно трактовать как no-op, если уже нет |
| Компания | `DELETE_COMPANY` из `OnBeforeCrmCompanyDelete` (B24) → сайт (`Company::deleteCompanyElement`) | Повтор `DELETE_COMPANY` должен оставаться no-op при уже удалённом элементе ИБ |
| Пользователь при регистрации | `ACTIVE=N` + запись ID контакта в `UF_BITRIX24_ID` и `UF_B24_USER_ID` (`RegisterUserCompany`) | см. код |

Дополнительно: заголовок **`X-Idempotency-Key`** (опционально) может использоваться для дедупликации повторных POST с CRM — реализация по мере необходимости; до внедрения полагаться на идемпотентность обработчиков `ACTION`.

## 3. Транспорт сайт → CRM (`to-crm`)

Помимо `URL_B24` и обёрток зафиксировать **заголовки решений** (детали реализации — в коде):

- **Аутентификация** к порталу: вебхук-приложение, OAuth, пользовательский контекст, служебный пользователь — что принято в проекте.
- **Ошибки и повторы:** поведение при HTTP 5xx, таймаутах, лимитах API (retry, backoff, когда не повторять 4xx кроме отдельных кодов).
- **Окружения:** на сайте `URL_B24` можно задать через переменную окружения `URL_B24` (см. `local/php_interface/init.php`); на портале базовый URL сайта — `EKLEKTIKA_SITE_URL`, секрет для входящего `ajax` — `EKLEKTIKA_SYNC_INBOUND_SECRET` (совпадает с `local/sync/config.local.php` на сайте).
- TLS verify в `RestClient`: по умолчанию включен для HTTPS-хостов, отключается для `localhost/127.0.0.1`; принудительное поведение задаётся через env `B24_REST_TLS_VERIFY` (`true|false`).

## 4. `ajax.php` и `from-crm/`: владение кодом

**Целевое правило:** `local/classes/ajax.php` — **тонкий фасад**: проверка доступа/подписи, маршрутизация по `ACTION`, делегирование в классы канала **`sync/from-crm/`** (или эквивалент после переноса).

- Логика применения данных к сущностям сайта **не** размазывается бесконечно по `ajax.php` в обход `from-crm/`.
- Критерий готовности этапа рефакторинга: новые обработчики входящих `ACTION` добавляются в `from-crm/`, фасад только вызывает их.
- Статус на 2026-04-24: правило реализовано через `InboundGateway::dispatch($_REQUEST)` + `InboundSecurity::assertInboundAllowed()` в `ajax.php`.
- Для `UPDATE_COMPANY` и `DELETE_COMPANY` шлюз возвращает JSON c `success` и `data` (в т.ч. `deleted`), а для неизвестного `ACTION` — HTTP `400` и `unknown_action`.

До переноса наследие может нарушать это правило — отражать в плане миграции явным шагом.

Для регистрации через `personal/ajax/ajax-register-action.php` silent catch запрещён: sync-ошибка должна возвращаться как контролируемый `success=false`, без авторизации пользователя и без «тихого» продолжения сценария.

## 5. Недоступность сайта и повторы на стороне B24

Идемпотентность на **приёме** на сайте — одна часть. Отдельно описать поведение **отправителя** (CRM):

- Ответ сайта **5xx**, **timeout**, обрыв сети — **retry**, **backoff**, ограничение числа попыток.
- Нужна ли **дедупликация** на стороне CRM (чтобы не плодить дубли при повторной отправке того же логического события).
- Связь с очередью/планировщиком на портале, если появится.
- Статус на 2026-04-24: в `OutboundRequest` реализованы повторы только для `HTTP 429/503` с backoff и jitter; ошибки cURL и другие HTTP-коды возвращаются как `success=0`.
- Безопасность входящего канала: при наличии `EKLEKTIKA_SITE_SYNC_TOKEN` B24 отправляет и `sync_token` в POST, и заголовок `X-Sync-Token`.

### Сохранение компании в CRM: `UPDATE_COMPANY` и цепочка `UPDATE_CONTACT`

При изменении **компании** в Bitrix24 обработчик на портале часто **обходит привязанные контакты** и для каждого шлёт на сайт **`UPDATE_CONTACT`** (ФИО, маркетинг, UF и т.д.). В логе `b24-to-site-sync.log` это выглядит как сохранение компании «с точки зрения пользователя», но в **`Params` всё равно будет `ACTION => UPDATE_CONTACT`**: так задумано на стороне отправителя, это не замена обновления карточки компании.

**Свойства компании** на сайте (ИБ, `OS_COMPANY_*`) меняются **только** отдельным POST с **`ACTION => UPDATE_COMPANY`** и полями компании (`OS_COMPANY_B24_ID` и остальное из контракта). Если при сохранении компании в логе **нет** такого запроса в пределах нескольких секунд, сайт компанию не обновит — нужно править цепочку в **`eklektika-ru-b24`** (например `CompanySync` / `onAfterCrmCompanyUpdate`): явно вызывать тот же транспорт, что и для контактов, с `UPDATE_COMPANY` и нужным payload.

**Проверка:** по времени сохранения компании в логе найти блок с `ACTION => UPDATE_COMPANY`. Если его нет — баг или недореализация на портале; если есть — смотреть HTTP/JSON ответа сайта на этот POST.

## 7. (Опционально) Лимиты и «куда смотреть при аварии»

**Не обязательный** слой для конвейера обмена: пока всё едет, сюда часто не возвращаются. Имеет смысл заполнить **по желанию** или когда всплывёт боль:

- крупные payload / таймауты HTTP — при необходимости одна строка + связь с §3;
- короткая шпаргалка «обмен встал» — лучше в **[runbook.md](runbook.md)** (на полстраницы: endpoint, константы URL, кто шлёт `Updater`), без метрик и дашбордов как требования.

Конвейер (сущности, ключи, безопасность входа, идемпотентность, конфликты) — в §1–6, не в §7.
