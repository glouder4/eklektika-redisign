## Цель
Скорректировать интерпретацию результатов на этапах property-фильтра и exact-проверки, чтобы ложноположительные кандидаты не трактовались как дубли.

## Объем работ
- Уточнить правила перехода между этапами `candidates -> exact -> decision`.
- Привести финальную ветку решения к однозначному статусу (`exact_match_found` или `no_exact_match`).
- Исключить сценарии, где промежуточные кандидаты влияют на финальный диагноз как «дубль».
- Обеспечить совместимость с текущим контрактом данных ИБ/ИНН.

## Входы
- Контракт событий и baseline из `01-baseline-and-logging-contract`.
- Текущая реализация резолвера ИНН в ajax-регистрации.

## Выходы
- Обновленная логика резолва, синхронизированная с целевой семантикой.
- Явно определенные граничные случаи (кандидаты есть, exact-match нет).

## Зависимости
- Согласованный словарь событий из подзадачи 01.
- Наличие тестового/воспроизводимого набора кейсов по ИНН.

## Риски
- Регрессия в соседних сценариях повторного использования резолвера.
- Скрытая зависимость от формата входных свойств или нормализации ИНН.

## Индивидуальные критерии готовности
- Промежуточный `candidate_count` не определяет финальный результат сам по себе.
- Ветка `no_exact_match` корректно отрабатывает без ложного статуса «дубль».
- Граничные кейсы описаны и покрыты проверками из подзадачи 03.

## Чеклист проверки результата
- [ ] Сценарий с `candidate_count>0` и без exact-match завершается как `no_match`.
- [ ] Сценарий с exact-match детерминированно завершается как `match/duplicate`.
- [ ] Нет противоречия между финальным статусом и emitted events.
- [ ] Обновлен статус подзадачи и синхронизирован с `task.md`.
