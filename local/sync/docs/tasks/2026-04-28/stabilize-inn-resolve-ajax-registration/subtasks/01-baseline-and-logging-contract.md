# Subtask 01: Baseline and logging contract

## Родительская задача
- [task.md](../task.md)

## Статус
- `todo`

## Цель
Зафиксировать текущую цепочку резолва ИНН и определить однозначный контракт событий логирования по этапам, чтобы исключить неверную трактовку кандидатов как дублей.

## Объем работ
- Собрать baseline по текущим событиям резолва в ajax-регистрации.
- Описать этапы: property-кандидаты, exact-проверка, финальное решение.
- Зафиксировать, какие события считаются обязательными, какие являются шумом.
- Утвердить ожидаемый словарь событий и минимальный набор контекстных полей.

## Входы
- Логи: `local/logs/ajax-register-action.log`.
- Код/контракт интеграции: `local/sync/docs/functional-contract.md`.
- Реализация ajax-регистрации и резолва ИНН (владелец определяется на этапе baseline).

## Выходы
- Карта событий «как есть» и «как должно быть».
- Список событий к сохранению/переименованию/удалению.
- Критерии диагностической корректности для последующих подзадач.

## Зависимости
- Доступ к актуальным логам и воспроизводимому кейсу.
- Подтверждение продуктового смысла терминов `duplicate` и `no_match`.

## Риски
- Неполный baseline из-за непокрытых веток выполнения.
- Подмена бизнес-терминов техническими деталями без согласования.

## Индивидуальные критерии готовности
- Зафиксированы все ключевые этапы резолва и ожидаемые исходы по каждому этапу.
- Для каждого события определены: когда эмитится, зачем нужно, какие поля обязательно содержит.
- Шумные/двусмысленные события помечены как кандидаты на удаление/замену.

- [ ] Есть связка между симптомом в логах и конкретным этапом пайплайна.
- [ ] Есть таблица или список целевых событий без смысловых пересечений.
- [ ] Сформулированы правила, отделяющие `candidate_count` от факта дубля.
