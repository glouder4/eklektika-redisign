# Task: Stabilize INN resolve in ajax registration

## Бизнес-цель
Стабилизировать резолв юрлица по ИНН в `ajax-регистрации`, чтобы убрать ложную интерпретацию дублей и сделать диагностические события в логах однозначными для поддержки и разработки.

## Контекст и наблюдаемый симптом
- По логу `local/logs/ajax-register-action.log` фиксируется событие `resolve_site_company_by_inn_none_after_props` с `candidate_count=3`.
- Это указывает на ложноположительные кандидаты на этапе property-фильтрации, которые затем отбрасываются exact-проверкой.
- Итоговое поведение и телеметрия сейчас выглядят как «дубли есть», хотя по факту совпадений нет.

## Scope
- Уточнить контракт этапов резолва ИНН: `property candidates -> exact check -> final decision`.
- Нормализовать семантику событий логирования по этапам (что считается кандидатом, матчем, отбраковкой, дублем).
- Убрать шумные/двусмысленные события и оставить только полезные для диагностики сигналы.
- Зафиксировать минимальный набор регрессионных проверок для сценария ajax-регистрации.

## Non-goals
- Не менять бизнес-правила регистрации пользователя и активации аккаунта.
- Не выполнять рефакторинг всего CRM/site обмена вне участка резолва ИНН.
- Не менять контракт внешних интеграционных endpoint'ов, если это не требуется для исправления ложных дублей.

## Definition of Ready (DoR)
- Подтвержден симптом в логах с примерами событий и контекстом (`inn_hash`, `candidate_count`).
- Определен владелец участка кода резолва ИНН и точки логирования.
- Согласована целевая семантика событий: что означает `candidate`, `exact_match`, `duplicate`, `none`.
- Согласован формат приемки по логам и сценариям (какие события должны/не должны появляться).

## Definition of Done (DoD)
- Ложноположительные «дубли» после property-фильтра больше не интерпретируются как реальные совпадения.
- События логирования отражают фактический исход по этапам и не противоречат друг другу.
- Для входного кейса с ранее шумным `candidate_count>0` финальная диагностика явно показывает отсутствие exact-match (без misleading event).
- Подготовлен и пройден чеклист регрессионной проверки ajax-регистрации.
- Документация задачи и подзадач актуализирована, статусы обновлены.

## Риски и зависимости
- Риск скрыть полезный сигнал при чрезмерной фильтрации логов.
- Риск регрессии в ветках, где используется тот же резолвер ИНН в других сценариях.
- Зависимость от корректности данных свойств ИБ и нормализации ИНН до этапа exact-проверки.
- Зависимость от текущего функционального контракта обмена в `local/sync/docs/functional-contract.md`.
