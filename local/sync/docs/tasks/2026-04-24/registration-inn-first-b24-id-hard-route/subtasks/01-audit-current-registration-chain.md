# Subtask 01: Аудит текущей цепочки регистрации (frontend + ajax + usersync + rest + b24 endpoint)

## Родительская задача
- `../task.md`

## Цель
Построить подтвержденную карту текущего сценария регистрации и определить точки, где выполняется поиск/создание компании и как используется (или не используется) связь `OS_COMPANY_B24_ID`.

## Объем работ
- Проанализировать текущий flow от frontend-инициации регистрации до завершения интеграционных вызовов.
- Зафиксировать все места, где используется ИНН и/или B24 company ID.
- Выявить ветки, где при наличии локальной связи выполняется повторный поиск по ИНН в B24.
- Подготовить таблицу расхождений as-is против целевого требования.

## Входы
- Код и конфигурации в `bitrix-docker/www`.
- Релевантные интеграционные участки в `eklektika-ru-b24`.
- Контракт `local/sync/docs/functional-contract.md`.

## Выходы
- Документированная диаграмма/описание текущей цепочки.
- Список точек изменения и рисков регрессии.
- Перечень расхождений с приоритетами (critical/high/medium).

## Зависимости
- Доступность всех участков цепочки в репозиториях.
- Валидный mapping сущностей ИБ 23 <-> B24 Company.

## Риски
- Скрытые side-effects в legacy-обработчиках Bitrix-событий.
- Разная логика в дублях классов/оберток transport-слоя.

## Индивидуальные критерии готовности
- Карта цепочки покрывает все обязательные узлы: frontend, ajax, usersync, rest, b24 endpoint.
- Для каждого узла зафиксировано условие перехода и ключевые входные/выходные поля.
- Подтверждено минимум одно место текущего отклонения от правила B24-by-ID (или явно указано, что отклонений нет).
- Результат согласован с `go-teamlead`.

## Чеклист проверки результата
- [ ] Все участки цепочки перечислены и связаны по порядку выполнения.
- [ ] Указаны конкретные поля: ИНН, `OS_COMPANY_B24_ID`, связанный ID ИБ 23.
- [ ] Выделены места поиска по ИНН в B24 и условия их вызова.
- [ ] Зафиксированы риски и влияние на downstream-сценарии.
