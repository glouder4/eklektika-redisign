# Subtask 04: Тест-план и ручной чеклист

## Родительская задача
- `../task.md`

## Цель
Подготовить проверяемый тест-план для нового контракта регистрации и обеспечить воспроизводимую ручную приемку до и после релиза.

## Объем работ
- Сформировать матрицу тест-сценариев:
  - happy-path с существующей локальной связью `OS_COMPANY_B24_ID`;
  - сценарии без локальной связи (допустимый INN lookup в B24);
  - edge-cases: битая связь, дубликаты, частичные ошибки B24.
- Определить входные наборы данных для каждого сценария.
- Подготовить ручной чеклист в порядке выполнения и ожидаемых результатов.
- Задать критерии pass/fail для QA gate.

## Входы
- Контракт to-be из Subtask 02.
- Реализационный план и миграционные правила из Subtask 03.
- Технические ограничения тестовой среды.

## Выходы
- Тест-план с покрытием функциональных и регрессионных рисков.
- Ручной чеклист для приемки релиза.
- Критерии для `qa-lead` verdict.

## Зависимости
- Готовность тестового контура и тестовых учетных данных.
- Доступ к логам регистрации и B24 REST-вызовов.

## Риски
- Недостаточное покрытие редких исторических кейсов.
- Невоспроизводимость race-condition без нагрузочного контура.

## Индивидуальные критерии готовности
- Тест-план содержит минимум: 1 happy-path + 5 негативных/edge сценариев.
- Для каждого сценария указаны preconditions, шаги, ожидаемый результат.
- Проверка "нет повторного поиска по ИНН в B24 при наличии `OS_COMPANY_B24_ID`" выделена отдельным обязательным тестом.
- Пакет принят `qa-lead` как достаточный для quality gate.

## Чеклист проверки результата
- [ ] Есть явная трассировка тестов к требованиям контракта.
- [ ] Описаны критерии завершения тестирования и формат дефект-репорта.
- [ ] Включены проверки логов/трейсов по ключевым веткам.
- [ ] Определены блокирующие и неблокирующие дефекты.
