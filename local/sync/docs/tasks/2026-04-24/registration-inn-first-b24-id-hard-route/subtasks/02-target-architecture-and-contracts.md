# Subtask 02: Целевая архитектура и контракты (локальный INN-first + B24-by-ID)

## Родительская задача
- `../task.md`

## Цель
Сформировать и согласовать целевой архитектурный контракт регистрации, где приоритет отдается локальной проверке по ИНН в ИБ 23, а при наличии `OS_COMPANY_B24_ID` все операции в B24 выполняются строго по этому ID.

## Объем работ
- Описать целевой алгоритм ветвления:
  - локальная проверка ИНН в ИБ 23;
  - ветка "связь есть" -> только B24-by-ID;
  - ветка "связи нет" -> разрешен поиск по ИНН в B24;
  - fallback-поведение при битой связи.
- Зафиксировать контракт входных/выходных данных между слоями (ajax/usersync/rest).
- Определить правила идемпотентности и предотвращения дублей.
- Зафиксировать ограничения безопасности и аудита вызовов.

## Входы
- Результаты Subtask 01 (as-is аудит).
- Текущий `functional-contract.md`.
- Бизнес-требование по приоритету локальной связи.

## Выходы
- Согласованный to-be контракт потока регистрации.
- Явные pre/post-conditions для ключевых этапов.
- Список обязательных инвариантов для реализации и тестирования.

## Зависимости
- Решения `go-teamlead` по спорным архитектурным вопросам.
- Подтверждение совместимости с текущими endpoint-ограничениями портала.

## Риски
- Недоопределенные ветки fallback при невалидном `OS_COMPANY_B24_ID`.
- Потеря обратной совместимости для редких исторических кейсов.

## Индивидуальные критерии готовности
- Контракт содержит формальную схему ветвления и критерии выбора ветки.
- Явно зафиксировано правило: поиск по ИНН в B24 запрещен при валидной локальной связи.
- Определены инварианты данных и точка истины для company linkage.
- Контракт принят `go-teamlead` как база для реализации.

- [ ] Есть список обязательных полей и проверок целостности.
