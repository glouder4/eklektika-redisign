# Task: Переписывание сценария регистрации (INN-first на сайте, B24-by-ID при наличии связи)

## Бизнес-цель
Стабилизировать и унифицировать сценарий регистрации пользователя так, чтобы первичным источником истины по компании был сайт (ИБ 23), а взаимодействие с B24 выполнялось по `OS_COMPANY_B24_ID`, если эта связь уже известна. Это снижает риск дублирования компаний и рассинхронизации при повторных регистрациях.

## Контекст и цель
Требуется полный hard-маршрут для переписывания логики регистрации в проекте `eklektika-ru` (сайт) и проверка сквозной интеграционной цепочки с `eklektika-ru-b24` (портал). Ключевая функциональная норма: поиск компании в B24 по ИНН разрешен только при отсутствии локальной связи в ИБ 23.

## Scope
- Полный аудит текущего потока регистрации: frontend -> `personal/ajax/ajax-register-action.php` -> usersync/rest модули -> endpoint и обработчики B24.
- Формализация целевого архитектурного контракта:
  - шаг 1: проверка компании по ИНН в локальной базе (ИБ 23);
  - шаг 2: если есть `OS_COMPANY_B24_ID`, все операции в B24 строго по этому ID;
  - шаг 3: поиск/создание по ИНН в B24 допускается только если локальная связь отсутствует.
- Подготовка плана реализации и миграционных оговорок (без выполнения кода в этом task-artifact).
- Подготовка тест-плана и ручного чеклиста.
- План обновления `local/sync/docs` и контрактной документации.

## Non-goals
- Не выполнять кодовые изменения в рамках данного документационного пакета.
- Не выполнять миграции данных или массовые backfill-операции.
- Не утверждать финальные технические решения за `go-teamlead` (роль разработки и код-аудита).

## Definition of Ready (DoR)
- Подтверждено, что сценарий относится к уровню `hard` (кросс-подсистемная интеграция + контрактные изменения + регрессионные риски).
- Согласованы границы цепочки для аудита: frontend, ajax, usersync, rest, B24 endpoint.
- Доступны действующие источники требований:
  - `bitrix-docker/www` (продуктовый код и docs),
  - `bitrix-docker/www/local/sync/docs/functional-contract.md` (приоритетный контракт),
  - релевантные участки `eklektika-ru-b24`.
- Назначены ответственные роли hard-маршрута:
  - `go-teamlead` (техническое руководство/аудит),
  - `go-senior-developer-executor` (реализация),
  - `qa-lead`, `security-reviewer`, `release-manager`, `sre-operations-reviewer`.

## Definition of Done (DoD)
- Все подзадачи из раздела "Подзадачи" переведены в `done` либо формально закрыты с зафиксированным `blocked` и планом снятия блокера.
- Целевой контракт регистрации (INN-first local + B24-by-ID) описан и принят `go-teamlead`.
- Подготовлен и выполнен минимальный набор проверок:
  - функциональные проверки happy-path и edge-cases;
  - негативные проверки некорректных/неполных связей;
  - верификация отсутствия повторного поиска по ИНН в B24 при наличии `OS_COMPANY_B24_ID`.
- Обновлена документация в `local/sync/docs` с явной фиксацией новых правил и переходных ограничений.
- Получены вердикты hard-gates: QA, Security, Release, SRE.

## Риски
- Несогласованность текущей бизнес-логики между legacy-классами и новыми модулями.
- Появление дубликатов компаний в B24 из-за конкурентных регистраций.
- Неполные/битые связи `OS_COMPANY_B24_ID` в исторических данных.
- Регрессии в связанных сценариях: создание контакта, обновление реквизитов, события синхронизации.
- Разночтения между кодом и устаревшими заметками (требуется эскалация при обнаружении).

## Зависимости

- Доступ к коду сайта (`bitrix-docker/www`) и портала (`eklektika-ru-b24`) для сквозного аудита.
- Решение `go-teamlead` по миграционной стратегии данных с отсутствующей/некорректной связью.
- Тестовый стенд/данные для проверки сценариев регистрации и интеграционных вызовов.
