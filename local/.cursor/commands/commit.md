# commit

Сделай коммит(ы) последних изменений проекта **автономно**, даже если пользователь не передал дополнительный контекст.

## Цель команды

- автоматически собрать контекст изменений;
- разложить изменения на логические группы;
- выполнить безопасные коммиты по правилам проекта;
- вернуть пользователю прозрачный итог (что и как закоммичено).

## Обязательный способ исполнения

1. Собери входной git-контекст (в параллель):
   - `git status --short`
   - `git diff -- .`
   - `git log --oneline -10`
2. Выполни авто-группировку изменений в логические коммиты (по умолчанию, если пользователь явно не просил "один коммит"):
   - приоритет групп:
     1) code/feature changes;
     2) rules/agents/skills/commands process changes;
     3) documentation;
     4) infra/support files.
   - не смешивай несвязанные изменения в один коммит.
3. Для **каждой** группы выполни последовательность:
   - `git add` только файлов текущей группы;
   - создай временный файл сообщения `.git-commit-msg.tmp.txt` в корне репо;
   - запиши сообщение коммита на русском языке, UTF-8 (без BOM);
   - перед коммитом обязательно проверь содержимое `.git-commit-msg.tmp.txt` и удали любые сервисные подписи/трейлеры (`Made-with: Cursor`, `Co-authored-by:`, `Generated by`, и аналоги);
   - если после проверки остались такие строки — **не выполняй commit**, сначала очисти файл сообщения;
   - выполни `git commit -F .git-commit-msg.tmp.txt`;
   - проверь `git log -1 --pretty=%B` и убедись, что в финальном сообщении нет сервисных трейлеров;
   - удали временный файл;
   - проверь `git status --short`.
4. Если commit упал из-за shell/parser/encoding:
   - повтори шаг через PowerShell-safe синтаксис (`;`, без heredoc и без `&&`);
   - снова проверь `git status --short` перед следующей попыткой.

## Жесткие правила (не нарушать)

- Всегда использовать сообщения коммита на русском языке (если пользователь явно не попросил English).
- Не использовать `git commit -m` для финального коммита: только через временный `.txt` файл.
- Не добавлять сервисные трейлеры в сообщение (`Made-with: Cursor` и аналоги).
- Если сервисный трейлер обнаружен в тексте сообщения, commit с таким сообщением запрещен до полной очистки.
- Не использовать destructive git-команды (`reset --hard`, `checkout --`, force push и т.п.).
- Не использовать `--no-verify`, `--amend` и другие обходы без явной просьбы пользователя.
- Не пушить в remote, если пользователь не просил.

## Когда нужен один коммит

- Если пользователь явно сказал "сделай один коммит" / "в один коммит" — допускается один коммит.
- Иначе по умолчанию делать несколько логических коммитов по группам выше.

## Финальный ответ пользователю

- какие группы были сформированы;
- какие файлы вошли в каждый коммит;
- commit hash и message каждого созданного коммита;
- остались ли незакоммиченные изменения.
