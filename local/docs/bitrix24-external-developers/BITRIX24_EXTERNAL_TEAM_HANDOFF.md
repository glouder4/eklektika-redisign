# Пакет для команды Bitrix24 (внешняя передача)

**Назначение:** один входной документ: что настроить на портале Bitrix24, куда слать HTTP-запросы на сайт и какие файлы приложить к переписке. Детальные таблицы полей — в приложенных документах ниже.

**Дата пакета:** 2026-04-29  
**Контур интеграции:** модули `yomerch.*` на сайте (PHP/Bitrix).

## 2. Куда слать запросы с портала Bitrix24 на сайт

**Канонический URL (относительно публичного домена сайта):**

```text
https://<ВАШ_ДОМЕН>/local/modules/yomerch.b24.inbound/endpoint.php
```

**Метод:** `POST`  
**Тело:** `application/x-www-form-urlencoded` (поля как параметры формы; сервер читает `$_REQUEST`).

**Обязательно в каждом запросе:**

| Элемент | Описание |
|---------|----------|
| Поле `ACTION` | Одна из команд из §3 |
| Аутентификация | Секрет сайта передаётся как **`X-SYNC-TOKEN`** (рекомендуется) или поле **`sync_token`** в теле (если на сайте не включён режим «только заголовок») |
| Идемпотентность | Рекомендуется заголовок **`X-Sync-Request-Id`** **или** поле **`REQUEST_ID`** — один и тот же ID при повторной доставке одного события; при включённом dedup на сайте повтор в окне TTL вернёт **409** (см. §5) |

**Ответ:** заголовок **`X-Sync-Request-Id`** дублирует присвоенный/принятый идентификатор запроса (удобно для корреляции в логах B24 и сайта).

**Опционально (если на сайте включено в `config.local.php`):**

| Заголовок / поле | Назначение |
|------------------|------------|
| `X-Sync-Timestamp` или `sync_ts` | Unix-время; при `inbound_max_skew_seconds > 0` проверяется расхождение с часами сервера |
| `X-Sync-Signature` | HMAC-SHA256 от **сырого тела** запроса (`php://input`), если задан `inbound_hmac_secret` |
| `require_post_method` / `inbound_require_header_token` | Жёсткий режим: только POST и/или только токен в заголовке (профиль `strict` на стороне сайта) |

Подробности безопасности и fail-closed: раздел **2.1** в [b24_site_contracts_yomerch.md](./b24_site_contracts_yomerch.md).

## 3. Список ACTION (входящие команды CRM → сайт)

| ACTION | Назначение (кратко) |
|--------|---------------------|
| `UPDATE_GROUP` | Синхронизация группы пользователей |
| `UPDATE_CONTACT` | Обновление пользователя сайта по контакту B24 |
| `UPDATE_BATCH_USERS` | Пакетное обновление признака (напр. маркетинг-агент) |
| `DELETE_CONTACT` | Удаление пользователя по контакту |
| `DELETE_COMPANY` | Удаление компании на сайте |
| `UPDATE_COMPANY` | Обновление карточки компании на сайте |
| `SYNC_COMPANY_CONTACTS` | Синхронизация состава контактов компании |
| `UPDATE_MANAGER` | Обновление привязки менеджера |

Обязательные поля по каждому ACTION — в [b24_site_contracts_yomerch.md](./b24_site_contracts_yomerch.md), раздел **2.3**.

## 4. Коды ошибок на входе (до бизнес-логики)

Ответы в формате **JSON**, HTTP **403** (если не указано иначе):

| `error` / `error_code` | Когда |
|------------------------|--------|
| `sync_forbidden` | Нет секрета в конфиге (кроме явного dev-override), неверный токен, запрещённый метод |
| `sync_method_not_allowed` | Не POST при включённом `require_post_method` |
| `sync_timestamp_invalid` | Нет или неверный timestamp при проверке skew |
| `sync_timestamp_expired` | Расхождение времени больше лимита |
| `sync_signature_missing` | Включён HMAC, подпись не передана |
| `sync_signature_invalid` | Подпись не совпала с телом |

После прохождения security возможны ответы валидатора/gateway (неверный payload, неизвестный ACTION, блокировка политикой, дубликат) — см. основной контрактный документ и карту событий в §7.

## 5. Дедупликация (важно для роботов и повторов)

Если на сайте **`inbound_dedup_ttl_seconds` > 0** и передан **`REQUEST_ID` / `X-Sync-Request-Id`**:

- первый запрос с данным ключом и `ACTION` обрабатывается;
- повтор в пределах TTL → **HTTP 409**, тело с `error: duplicate_request`, `reason_code: dedup_duplicate`.

**Рекомендация для Bitrix24:** на каждое логическое событие CRM генерировать **стабильный** `REQUEST_ID` (или передавать ID события CRM), чтобы повторная доставка не создавала двойных эффектов на сайте.

## 6. Исходящий контур (сайт → Bitrix24)

Сайт вызывает REST портала через вебхук (`RestClient`). Список методов и полей — раздел **1)** в [b24_site_contracts_yomerch.md](./b24_site_contracts_yomerch.md). Конфигурация URL и токенов вебхуков на стороне сайта — файл **`b24_integration_config.php`** (шаблон без секретов: `b24_integration_config.example.php`); в переписку с B24 обычно достаточно подтвердить, что **исходящий вебхук** на портале имеет права на перечисленные методы.

## 7. Обновление таблицы ACTION → event / reason

После изменений кода на сайте таблицу можно пересобрать:

```bash
php local/modules/yomerch.b24.inbound/tools/export_contract_map.php docs/refactoring/generated_inbound_action_contract_map.md
```

Команда выполняется **на стороне репозитория сайта**; в эту папку актуальную версию `generated_inbound_action_contract_map.md` копируют перед отправкой внешней команде. На Windows без `php` в PATH: `pwsh local/modules/yomerch.b24.inbound/tools/run_contract_harness.ps1` (для прогона тестов контракта); для export — `wsl` + `php`, как в команде выше.

## 8. Чеклист для настройщика Bitrix24

- [ ] Исходящий запрос (робот / бизнес-процесс / приложение) настроен на **POST** указанного URL.
- [ ] Передаётся **`ACTION`** и все **обязательные поля** по таблицам контракта.
- [ ] Настроена передача **секрета** (`X-SYNC-TOKEN` или `sync_token`) согласованно с владельцем сайта.
- [ ] Для идемпотентности: передаётся **`X-Sync-Request-Id`** или **`REQUEST_ID`** (стабильный на повторы).
- [ ] Согласовано включение **HMAC** и/или **timestamp skew**, если на сайте включены соответствующие поля в конфиге.
- [ ] После изменений на портале — короткий **smoke-тест** по одному безопасному ACTION с записью `request_id` для сверки логов.
