# Initiative: регистрация при совпадении ИНН с уже заведённой на сайте компанией

## Policy

- **ИНН компании** совпадает с элементом каталога на сайте → регистрацию **не** блокируем; в `RegisterUserCompany::createB24Company()` выполняется merge в **`OS_COMPANY_USERS`**, в CRM создаётся **новый контакт** (`crm.contact.add`) и привязка к компании (`crm.contact.company.add`), см. `Company::createCompanyElement()`.
- **Ошибка регистрации** при неуникальной **личности**: контакт в CRM (`User::isUserRegistered` / `getContactID`). Дубль на сайте по LOGIN/EMAIL перехватывает ядро при `CUser::Add` — отдельный предварительный запрос к `b_user` по email/телефону убран (ложные срабатывания из‑за ключей полей и форматов).

## Problem (historical)

Блокировка `duplicate_inn` при локальном дубликате ИНН и сообщение «Компания с указанным ИНН уже зарегистрирована…».

| T2 | При ИНН в CRM и элементе головной компании на сайте — сценарий филиала (`crm.company.add` + `OS_HOLDING_OF`) без конфликта с merge по локальному ИНН | done |
| T3 | Явный возврат ID контакта из `createB24Company` | done |

## Related site rules

- У головной компании (`OS_COMPANY_IS_HEAD_OF_HOLDING` = да) свойство `OS_HOLDING_OF` на сайте сбрасывается (`Company::updateCompanyElement` / создание элемента; при входящем UF головной — `CrmInboundUfMap::applyCompanyInboundHeadOfHoldingUfToSiteProperties`).

## Risks

- Два разных юрлица с тем же ИНН в CRM маловероятны; при сомнениях усиливать проверки на стороне CRM.
- Дубль email в CRM и отсутствие записи на сайте — по-прежнему отсекается через CRM до проверки сайта; рассинхрон форматов телефона может дать ложное «нет дубля» на сайте.
